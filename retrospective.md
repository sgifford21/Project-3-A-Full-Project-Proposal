# Project Plan  
## The Locker Room – Online Storefront Concept

---

## Project Overview

This project plan supports the development of a two-page website prototype for The Locker Room. The project uses planning concepts from *Project Management for the Unofficial Project Manager*, including risk identification, TAME risk response strategies, and a simple work breakdown structure.

---

## TAME Risk Analysis

| Risk | Impact | Probability | Risk Level | TAME Strategy | Response Plan |
|---|---:|---:|---:|---|---|
| Coding errors in HTML or CSS cause the site to display incorrectly | 4 | 3 | 12 | Mitigate | Build one section at a time, test pages in the browser, and validate that links and styling work before final submission. |
| GitHub Pages does not publish correctly or the live link does not work | 5 | 3 | 15 | Mitigate | Publish early, verify the live URL, and allow time to troubleshoot settings before the deadline. |
| Scope creep causes the project to become too large | 4 | 2 | 8 | Eliminate | Keep the project limited to two pages, one stylesheet, and no real e-commerce functionality. |
| Time management conflicts reduce time available for final edits | 4 | 3 | 12 | Accept/Mitigate | Use a simple task schedule, complete required files first, and save final polish for the end. |
| Design does not look professional enough | 3 | 2 | 6 | Mitigate | Use a consistent color palette, readable fonts, clear navigation, and a simple layout. |

---

## Work Breakdown Structure and Task Schedule

| Task | Description | Deliverable |
|---|---|---|
| 1. Repository Setup | Create or prepare the GitHub repository and organize files | Public GitHub repository |
| 2. Documentation Folder | Create the `docs/` folder and add planning files | `scope.md`, `plan.md`, `retrospective.md` |
| 3. Homepage Build | Create homepage content using semantic HTML | `index.html` |
| 4. About Page Build | Create second page with project and business background | `about.html` |
| 5. CSS Styling | Apply consistent colors, spacing, typography, and layout | `style.css` |
| 6. Navigation Testing | Ensure both pages link to each other correctly | Working navigation |
| 7. README Update | Explain the project and include live site/documentation links | `README.md` |
| 8. GitHub Pages Deployment | Publish site through GitHub Pages | Live website URL |
| 9. Final Review | Check file structure, spelling, links, and rubric alignment | Submission-ready repository |

## Commit Plan

| Commit | Commit Message | Purpose |
|---|---|---|
| Commit 1 | Setup docs folder with scope and plan | Shows initial organization and project planning |
| Commit 2 | Added index and about pages with navigation | Shows website structure and semantic HTML development |

---

## Success Criteria

The project will be successful if the repository includes all required documentation, contains a functional two-page website, uses one external CSS file, includes at least three meaningful commits, and is published through GitHub Pages.
